# 《工程教育专业认证智能服务系统》需求规格说明书 (SRS)

---

## 一、 引言

### 1.1 项目背景与建设意义
随着《华盛顿协议》全面落地，中国工程教育认证协会（CEEAA）对我国高校工科专业实施了全方位的工程教育认证。认证的核心在于 **OBE（Outcome-Based Education，成果导向教育）** 理念，即以“学生的毕业产出”反向设计培养方案、教学大纲及考核机制。

目前，高校在进行专业认证自评工作时，通常面临以下几大严峻挑战：
1. **数据收集孤岛化**：学生的平时成绩、实验报告、期末卷面成绩等散落在不同的纸质媒介或分散在各个教师的 Excel 账单中，缺乏统一归档和防篡改校验。
2. **支撑关系缺乏闭环**：培养目标对毕业要求、毕业要求对课程大纲、课程大纲对考核细项的“支撑矩阵”关系主要依靠人工口头规划，缺乏系统级的数量校验与合理性审计。
3. **达成度计算极其繁琐**：传统的直接达成度与间接达成度计算涉及复杂的二级/三级权重乘积、加权均值运算，人工利用 Excel 处理经常由于单元格公式错误导致自评数据失真。
4. **报告编写工作量繁重**：专业负责人不仅需要撰写长达数百页的《专业自评报告》，还要手动扫描和归档大量的高、中、低档“学生佐证材料样本”。

本系统基于 **PDCA（质量持续改进）** 循环开发，通过强大的直接与间接评价计算引擎以及 **Spring AI Agent** 工具链，全方位降低高校在专业认证过程中的填报与分析工作量，实现认证自评的规范化、流程化与智能化。

## 二、 系统架构与业务闭环模型

系统的核心业务逻辑严格遵循质量持续改进闭环，其控制流如下所示：

```mermaid
flowchart TD
    subgraph Plan (方案设计与指标映射)
        A[人才培养方案] -->|分解出| B[毕业要求指标点]
        B -->|构建| C[课程支撑权重矩阵]
        C -->|细化| D[课程目标 COs]
        D -->|建立| E[课程考核评价方案 Rubrics]
    end

    subgraph Do (教学运行与成绩采集)
        E -->|派发教学班| F[教学运行班级]
        F -->|多维度录入| G[学生平时与卷面细分成绩]
        F -->|教师归档| H[高/中/低档佐证材料样本]
    end

    subgraph Check (达成度评价引擎)
        G -->|计算| I[班级课程目标直接达成度]
        I -->|综合| J[专业毕业要求直接达成度]
        K[学生/毕业生/用人单位问卷] -->|加权计算| L[专业间接达成度]
    end

    subgraph Act (AI 赋能持续改进)
        J -->|JSON 输入| M[AI 诊断与改进建言 Agent]
        L -->|满意度分词| M
        M -->|输出改进方案| N[持续改进报告]
        N -->|反馈调整| A
    end
```

## 三、 角色定义与数据安全权限边界

系统基于严格的 **RBAC（基于角色的权限控制）** 架构设计，在物理和逻辑层面对敏感数据进行了高级别的权限隔离。

### 3.1 角色定义与操作矩阵

| 角色名称 | 业务系统定位 | 数据可见性边界 (安全红线) | 业务职责与核心操作 |
| :--- | :--- | :--- | :--- |
| **1. 学生** | 学习主体 | **个人数据物理隔离**。仅能查看自身在各门课程下的细分项得分，以及个人毕业要求雷达图。无权查看任何平行班级的他人成绩及专业全局大盘数据。 | 1. 查阅 OBE 导向的课程大纲与目标<br>2. 查看个人能力达成度雷达图及短板<br>3. 填写满意度及达成度间接调查问卷 |
| **2. 授课教师** | 教学执行与数据采集者 | **教学班级隔离**。仅有权管理自身所承接物理班级学生的明细成绩与佐证材料。无法查看其他班级的数据。 | 1. 录入基于课程目标维度的学生细分项分数<br>2. 按比例分类上传“高/中/低”档的试卷、作业 PDF 佐证材料<br>3. 针对班级达成度低分项在线撰写改进意见 |
| **3. 课程负责人** | 课程体系总设计师 | **课程级（跨班）可见**。可调阅其所负责课程下所有平行班的达成度汇总数据并进行横向质量对比。 | 1. 制定课程教学大纲与课程目标（COs）<br>2. 设定本课程的考核权重及细项映射矩阵（Rubrics）<br>3. 审核各班意见并汇总导出课程年度改进报告 |
| **4. 专业负责人** | 认证工作总指挥 | **全专业级可见**。可见本专业名下所有培养方案、课程体系、教师评分、班级达成度看板，并统筹自评报告编写。 | 1. 制定多版本人才培养方案并分解指标点<br>2. 审核“课程-毕业要求”支撑矩阵权重合理性<br>3. 派发自评报告章节编写任务，并自动套编数据表格 |

## 四、 核心功能模块详细规约

### 4.1 专业负责人控制台 (Program Director Console)

#### 4.1.1 培养方案与权重矩阵管理
1. **多版本克隆与历史修订**：系统支持培养方案的“版本基线管理”。专业负责人可以基于上一版培养方案一键克隆（如从 2020 版克隆到 2024 版），并在克隆体上微调指标点，且历史版本数据不会被覆盖。
2. **支撑权重合理性校验**：
   * 在配置“课程体系 - 毕业要求指标点”矩阵时，负责人为每门课程设定其支撑权重。
   * **强一致性校验**：系统会自动进行数学校验，某二级指标点下所有关联课程的权重之和必须严格等于 1.0（即 $\sum W_{support} = 1.0$）。如果不等于 1.0，系统会拦截发布并报警。

#### 4.1.2 毕业要求达成度计算引擎
1. **直接达成度加权计算**：
   系统一键抽取本培养方案名下所有关联课程的课程目标实际达成度数据，自动乘以该课程对毕业要求指标点的支撑权重，累加计算出最终的毕业要求指标点达成度值。
   $$R_{attain} = \sum_{j=1}^{m} C_j \cdot W_{support, j}$$
   *(其中 $C_j$ 为第 $j$ 门支撑课程的实际达成度，$W_{support, j}$ 为其对指标点的支撑权重)*。
2. **警示线监测**：系统默认设置 0.68 为达成度及格线。任何低于此值的能力项均以红色高亮显示，并自动列入“待改进问题清单”。

---

### 4.2 课程负责人控制台 (Course Coordinator Console)

#### 4.2.1 课程目标与支撑矩阵设定
1. **CO 设定**：负责人为本课程添加 3~5 个能力目标（Course Objectives）。
2. **指标点对齐**：为每条课程目标设定其对毕业要求二级指标点的支撑权重。

#### 4.2.2 考核大纲细化矩阵 (Rubrics 配置)
1. **成绩比例分配**：如“期末考试占 60%，平时作业占 20%，实验项目占 20%”。
2. **考核细项与目标强对齐**：
   * 支持到具体大题的分数分配。例如：期末考试卷面有 3 大题，第一题（20分）考查 CO1；第二题（40分）考查 CO2；第三题（40分）考查 CO3。
   * **输入校验**：各考核方式细分项满分乘以其对应大项的权重之和，必须等于本门课程的总分（100分）。

---

### 4.3 授课教师工作台 (Instructor Workstation)

#### 4.3.1 多维度成绩网格录入
1. **OBE 维度录入**：教师必须录入每个学生在每个预设考核细项下的得分，系统自动将其折算并归类到对应的课程目标中。
2. **格式导入校验**：支持通过 Excel 批量导入学生细项分数。导入时系统会自动验证学号唯一性、以及每项得分是否超出了满分上限。

#### 4.3.2 佐证材料电子化归档 (Evidence Material)
1. **高/中/低样本收集**：教师必须为每一个主要考核项（如期末考试）分别上传“优秀（High，成绩占前 10%）”、“中等（Medium，成绩占中间值）”、“不及格/边缘（Low，成绩在及格线附近）”三份真实学生答卷/报告的 PDF 扫描件。
2. **防遗漏提醒**：如果教师没有上传完整档次的佐证材料，系统会拦截本教学班的达成度最终提交，确保归档的完备性。

#### 4.3.3 班级直接达成度自动计算
系统自动基于本教学班全体学生的细项平均分计算各课程目标的达成度：
$$D_{direct} = \frac{\sum_{i=1}^{n} S_i \cdot W_i}{\sum_{i=1}^{n} T_i \cdot W_i}$$
*(其中 $S_i$ 为班级在考核细项 $i$ 下的平均得分，$T_i$ 为该细项满分，$W_i$ 为权重)*。

---

### 4.4 系统管理员后台 (Admin Panel)

#### 4.4.1 数据同步与集成接口
1. **教务基础数据同步**：支持标准 RESTful 接口对接高校选课和教务管理系统，一键同步当学期的师生名单、行政班级划分、教学班排课关联等基础数据。

#### 4.4.2 系统安全审计与操作日志
1. **防篡改审计记录**：任何修改已结课班级成绩、变更培养方案指标点权重、或者删除佐证材料文件的操作，均会被不可擦除的审计日志（`sys_audit_log` 表）实时捕捉。
2. **日志审计字段**：操作人 ID、登录账号、操作动作类型（LOGIN, CREATE, UPDATE, DELETE）、操作详情、客户端 IP、创建时间。

## 五、 🌟 Spring AI 接入与 AI Agent 协同方案设计

本系统通过集成 Spring AI，统一衔接主流大语言模型（如 DeepSeek-V3, GPT-4o），利用本地 pgvector（PostgreSQL）向量库进行 RAG（检索增强生成）知识检索，部署了 4 个具有独立职责的 AI Agent。

### 5.1 课程大纲设计助手 (Syllabus Agent)
*   **输入**：课程基本名称、教学主要大纲或各章节大体描述文本。
*   **RAG 知识检索**：Agent 读取负责人提供的文本，并在 pgvector 向量库中检索最新的《工程教育专业认证规范标准》以及同类专业优秀大纲库。
*   **AI 动作**：自动将大纲提炼并生成 3~5 条标准且符合 OBE 规范要求的“课程目标”文本（使用“阐述...”、“应用...为...建模”、“设计...的复杂架构”等具有可衡量层级的动作词汇），并智能推荐挂接最合理的毕业要求指标点。

### 5.2 试卷与课程目标对齐度审查 Agent (Exam Review Agent)
*   **输入**：教师上传的试卷扫描件 PDF 以及对应的课程大纲参数。
*   **AI 动作**：
    1. 调用 Spring AI 的多模态视觉接口，利用大模型对试卷内容进行 OCR 高精度识别。
    2. 精准拆分出各道大题考查的知识点和能力点。
    3. **对齐度审计**：对比该课程在大纲中定义的课程目标描述，对每一道题目进行交叉对比审计。
    4. **生成审计报告**：输出《期末试卷与课程目标一致性审查报告》，自动标注出试卷中“出题与目标偏离”或“题型考核难度与指标点认知度不符”的具体题目，提出改进警告。

### 5.3 持续改进建言 Agent (Continuous Improvement Agent)
*   **输入**：班级课程达成度原始数据 JSON（如 `{"CO1": 0.81, "CO2": 0.52}`）及学生在线填写的课程负面反馈分词词云。
*   **AI 动作**：
    1. Agent 进行诊断推理，识别达成度偏低（低于 0.68 警戒线）的课程目标（如 CO2 需求规格说明书撰写能力仅 0.52）。
    2. 融合分析学生词云中反映的痛点（如“规格书参考模板太旧”、“缺少练习时间”）。
    3. 结合内置的高等教育学与 OBE 改革案例库，自动生成具体的、有针对性的《持续改进方案草案》（如“CO2 达成度过低，建议将实验二课时增加 4 课时，在课堂中引入微型项目实践，提供业界标准的最新用例规格书模板，并提高作业中过程性评价的打分比重”），避免教师在反思撰写中敷衍塞责。

### 5.4 自评报告协同撰写 Agent (Self-Evaluation Report Agent)
*   **输入**：全专业的毕业要求达成度数据矩阵、各级对比趋势数据、毕业生反馈指标及用人单位满意度。
*   **AI 动作**：
    1. 使用 Spring AI 的 `Structured Outputs` 结构化生成接口，设定自评报告的 JSON schema 约束。
    2. 抽取大盘数据，大模型自动进行多维度的深度趋势分析与归因剖析。
    3. **自评报告套编**：自动将分析出的图表、统计表格和长篇学术分析文本插入到自评报告各章节模版中（如“第二章：毕业要求达成度分析”），大幅降低专业負責人撰写自评报告初稿的时间成本。

## 六、 技术栈与非功能性需求

### 6.1 开发技术架构

| 层面 | 技术选型 | 说明 |
| :--- | :--- | :--- |
| **基础开发语言** | Java 17 | 使用 LTS 版本的 JDK，拥有更好的垃圾回收和性能特性。 |
| **核心框架** | Spring Boot 3.2.0 | 现代 Java Web 开发的标准脚手架，开箱即用。 |
| **数据库** | MySQL 8.0 / PostgreSQL (pgvector) | MySQL 用于存储常规业务数据；PostgreSQL 的 pgvector 用于存储 AI Agent 切片知识的向量。 |
| **数据层组件** | MyBatis-Plus 3.5.5 | 提供强大的单表 CRUD 快捷开发，集成有分页拦截器。 |
| **AI 框架集成** | **Spring AI (1.0.0-M1+)** | 提供底层大模型调用的统一抽象和 RAG 框架集成。 |
| **前端架构** | Vue 3 + TypeScript + Pinia + Element Plus | 确保大盘驾驶舱界面的敏捷开发和高响应速度。 |

### 6.2 核心非功能性规约
1. **数据计算精度（绝对红线）**：
   * 所有涉及达成度计算、权重之和校验、学生实际得分的字段，在 Java 代码中**必须全部使用 `BigDecimal` 类型**（舍入模式使用 `RoundingMode.HALF_UP` 且保留两位小数）。
   * 严禁使用 `float` 或 `double` 进行计算以防浮点数精度丢失引发的账目差错。
2. **防篡改与不可物理删除日志**：
   * 系统的审计日志表 `sys_audit_log` 仅支持 `INSERT` 和 `SELECT` 权限。
   * 任何在应用程序级别尝试物理删除或修改日志的行为都必须被底层数据库拦截抛出异常。
3. **接口高并发与异步处理**：
   * 在期末大批量教师录入/导入成绩以及系统自动计算专业达成度雷达图时，系统应启用多线程或异步任务队列（如 Spring `@Async`）在后台处理运算，确保前端交互的响应时间 $\le 1.5$ 秒。
