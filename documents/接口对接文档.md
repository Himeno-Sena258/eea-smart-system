# 工程教育专业认证智能服务系统 - 接口对接文档

## 1. 文档说明

本文档用于前端与 Spring Boot 后端进行接口联调，接口设计依据项目功能需求、功能点截图以及 `documents/数据库.md` 中的数据库设计整理。

当前数据库文档中的核心表以 `sys_organization`、`sys_role`、`sys_user`、`class_info`、`student_info`、`program_scheme`、`grad_requirement`、`grad_indicator_point`、`course`、`course_objective`、`course_obj_indicator_map`、`teaching_class`、`teaching_class_student`、`assessment_method`、`assessment_item`、`student_score`、`evidence_material`、`course_attainment`、`continuous_improvement`、`survey_questionnaire`、`survey_answer`、`report`、`report_section`、`report_data_source`、`sys_audit_log` 等为准。若需求截图中的表名与数据库文档不完全一致，接口命名保持业务语义稳定，后端实现时按数据库文档映射。

## 2. 基础约定

### 2.1 接口前缀

```text
/api/v1
```

### 2.2 请求格式

- 普通请求：`Content-Type: application/json`
- 文件上传：`Content-Type: multipart/form-data`
- 批量导入：优先使用 `.xlsx` 文件，通过 `multipart/form-data` 上传
- 日期时间格式：`yyyy-MM-dd HH:mm:ss`
- 金额、分数、权重等小数：使用数字类型，权重范围通常为 `0.00 ~ 1.00`

### 2.3 登录校验约定

本项目后端采用简单登录校验方式，不使用 JWT。登录接口根据账号、密码查询数据库，校验通过后返回当前用户信息和角色信息。

前端可将当前登录用户信息保存到本地状态或浏览器存储中，用于页面路由守卫和按钮权限判断。退出登录时清除前端保存的用户信息即可。

若后端需要判断当前操作人，可在请求参数中显式传递 `operatorId`、`createdBy` 或 `userId`，具体字段以对应接口说明为准。

### 2.4 统一响应结构

```json
{
  "code": 0,
  "message": "success",
  "data": {},
  "traceId": "20260716190000123",
  "timestamp": "2026-07-16 19:00:00"
}
```

字段说明：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `code` | number | 业务状态码，`0` 表示成功 |
| `message` | string | 响应提示信息 |
| `data` | any | 响应数据 |
| `traceId` | string | 请求追踪 ID，用于排查问题 |
| `timestamp` | string | 服务端响应时间 |

### 2.5 分页响应结构

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "records": [],
    "pageNum": 1,
    "pageSize": 10,
    "total": 0,
    "pages": 0
  },
  "traceId": "20260716190000123",
  "timestamp": "2026-07-16 19:00:00"
}
```

分页查询通用参数：

| 参数 | 类型 | 必填 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| `pageNum` | number | 否 | `1` | 当前页 |
| `pageSize` | number | 否 | `10` | 每页数量 |
| `keyword` | string | 否 | - | 搜索关键词 |

| 方法 | 路径 | 说明 |

| `POST` | `/api/v1/auth/login` | 用户登录 |


### 4.2 用户开户规约

系统关闭普通用户自主注册，不提供 `/api/v1/auth/register`。所有学生学号、教师工号、行政班级归属及权限角色，均由系统管理员通过后台统一开户或导入官方花名册生成。

管理员开户接口见 [5.3 用户管理](#53-用户管理)。

```json

```

实现说明：后端如未维护服务端会话，可直接返回成功；前端负责清除本地保存的用户信息。

### 4.4.1 前端路由权限表

前端主导航与路由守卫以 `/api/v1/auth/me` 返回的 `roleCodes` 作为唯一权限来源。若用户拥有多个角色，前端只能在 `roleCodes` 范围内选择当前激活角色，不能提供任意模拟角色切换；切换激活角色后应重新计算可见菜单、页面模块与按钮权限。

| 前端路由 | 页面名称 | 开放角色 | 页面内差异化显示规则 | 数据库支撑情况 |
| --- | --- | --- | --- | --- |
| `/dashboard` | 角色工作台 | `ADMIN`、`DIRECTOR`、`COORDINATOR`、`INSTRUCTOR`、`STUDENT` | 按当前激活角色返回不同统计卡片和待办摘要 | 仅展示 [9.4 达成度总览看板](#94-达成度总览看板) 中现有数据库可支撑的字段 |
| `/admin/users` | 用户与组织管理 | `ADMIN` | 仅展示开户、组织、导入、审计等系统底座能力；不得展示业务成绩与达成度明细 | `sys_user`、`sys_role`、`sys_user_role`、`sys_organization`、`class_info`、`student_info`、`sys_audit_log` 支撑 |
| `/program` | 培养方案管理 | `DIRECTOR` | 专业负责人维护培养方案、毕业要求、指标点和课程支撑矩阵 | `program_scheme`、`grad_requirement`、`grad_indicator_point`、`course`、`course_obj_indicator_map` 支撑 |
| `/courses` | 课程质量管理 | `DIRECTOR`、`COORDINATOR`、`INSTRUCTOR`、`STUDENT` | `COORDINATOR` 可编辑课程目标与考核配置；`DIRECTOR`、`INSTRUCTOR`、`STUDENT` 默认只读查看与其权限相关的课程信息 | `course`、`course_objective`、`course_obj_indicator_map`、`assessment_method`、`assessment_item` 支撑 |
| `/teaching-classes` | 教学班与成绩 | `COORDINATOR`、`INSTRUCTOR`、`STUDENT` | `INSTRUCTOR` 可录入本教学班成绩和上传佐证；`COORDINATOR` 只能看其负责课程下教学班汇总；`STUDENT` 只能看本人明细 | `teaching_class`、`teaching_class_student`、`student_score`、`student_course_score`、`evidence_material` 支撑 |
| `/attainment` | 达成度分析 | `DIRECTOR`、`COORDINATOR`、`INSTRUCTOR`、`STUDENT` | `DIRECTOR` 看专业/年级级；`COORDINATOR` 看负责课程跨班汇总；`INSTRUCTOR` 看本人教学班；`STUDENT` 看个人，禁止返回他人成绩明细 | `course_attainment`、`grad_indicator_attainment`、`grad_requirement_attainment`、`student_score` 支撑；学生端班级均值需匿名聚合并控制最小样本量 |
| `/surveys` | 问卷与评价 | `DIRECTOR`、`STUDENT` | `DIRECTOR` 管理系统内问卷；`STUDENT` 填写本人可见问卷 | `survey_questionnaire`、`survey_answer` 支撑 |
| `/improvements` | 持续改进 | `DIRECTOR`、`COORDINATOR`、`INSTRUCTOR` | `INSTRUCTOR` 填写班级改进；`COORDINATOR`、`DIRECTOR` 查看可见范围内的改进记录 | `continuous_improvement` 支撑基础记录 |
| `/reports` | 自评报告协同 | `DIRECTOR`、被分派的 `INSTRUCTOR` | `DIRECTOR` 管理报告与章节分派；教师只查看和编辑 `assignedTo` 为自己的章节 | `report`、`report_section`、`report_data_source` 支撑 |
| `/profile` | 个人中心 | `ADMIN`、`DIRECTOR`、`COORDINATOR`、`INSTRUCTOR`、`STUDENT` | 展示本人资料、角色列表和账号安全信息 | `sys_user`、`sys_user_role`、`sys_role`、`sys_organization`、`student_info` 支撑 |

| `POST` | `/api/v1/admin/users` | 管理员开户 / 新增用户 |

| 参数 | 类型 | 必填 | 说明 |

```json
{
  "username": "2024001001",
  "password": "123456",
  "realName": "张三",

  "phone": "13800000000",
  "orgId": 10,
  "roleCodes": ["STUDENT"],
  "studentNo": "2024001001",
  "classId": 1
}
```

```json
{
  "id": 1001,
  "username": "2024001001",
  "realName": "张三",
  "email": "student@example.com",
  "phone": "13800000000",
  "status": 1,
  "orgId": 10,
  "orgName": "软件工程",
  "roleCodes": ["STUDENT"],
  "roleNames": ["学生"],
  "studentNo": "2024001001",
  "classId": 1,
  "className": "软件工程2024级1班",
  "createdAt": "2026-07-16 19:00:00"
}
```

启用或禁用用户请求参数：

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `status` | number | 是 | `1`-启用，`0`-禁用 |

重置密码请求参数：

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `newPassword` | string | 否 | 不传时默认重置为 `123456` |

实现说明：

- 普通用户不允许自主注册，用户由管理员开户或批量导入。
- 新增用户时使用 `roleCodes` 绑定角色，后端按角色编码写入 `sys_user_role`。
- 当 `roleCodes` 包含 `STUDENT` 时，`studentNo`、`classId` 必填，并同步写入 `student_info`。

| 字段 | 类型 | 含义 | 数据库支撑 |
| --- | --- | --- | --- |
| `stats.courseCount` | number | 本人选修课程数量 | `teaching_class_student` 联查 `teaching_class` 可统计 |
| `stats.averageTotalScore` | number | 本人课程总评均分 | `student_course_score.total_score` 可聚合 |
| `stats.passedCourseCount` | number | 已通过课程数量 | `student_course_score.is_passed` 可统计 |
| `todoItems.pendingSurveys` | object[] | 待填问卷 | `survey_questionnaire` 与 `survey_answer` 可判断是否提交 |
| `charts.classAverageComparison` | object[] | 匿名班级均值对比 | `student_score` 可聚合；不得返回其他学生明细，且建议设置最小样本量 |

实现说明：

- `dashboard/overview` 应按 `activeRole` 做数据隔离。`ADMIN` 不返回业务成绩与达成度明细；`STUDENT` 不返回任何其他学生的明细成绩。
- 前端原型中的固定数字（如活跃用户数、在库课程数、问卷回收率、报告进度）必须以后端响应为准；后端暂不支持的字段应返回 `null` 或不返回，由前端显示占位。
- 本节只定义现有数据库可直接统计或聚合出的字段。系统健康状态、失败登录风险、问卷目标范围、课程负责人归属、持续改进审核状态、个人达成度固化表等暂不进入接口返回，前端原型也不展示这些指标。

## 15. 前后端联调顺序建议

1. 基础数据：专业、培养方案、毕业要求、指标点、课程。
2. 课程大纲：课程目标、指标点支撑矩阵、考核方式、考核细项。
3. 教学运行：教师、学生、教学班、学生名单。
4. 成绩过程：成绩录入、过程记录、佐证材料上传。
5. 计算分析：课程目标达成度、毕业要求达成度、不达标学生筛查。
6. 评价闭环：问卷调查、持续改进、自评报告。
