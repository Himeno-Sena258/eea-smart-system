# 接口追加文档

本文用于追加 `documents/接口对接文档.md` 中遗漏的接口。检查范围包括：

- `documents/需求分析.md`
- `documents/系统角色功能表.md`
- `documents/接口对接文档.md`
- `frontend/src/services/*-service.ts`
- `documents/数据库.md`
- `documents/数据库变更记录.md`

追加原则：

- 只追加需求文档明确提到，且当前数据库结构可以支撑，或前端 service 已经存在但主接口文档漏写的接口。
- 当前数据库不支撑、数据库变更记录已移除、或主接口文档明确声明暂不开放的能力，不追加为正式接口，只在“暂不追加项”中说明。
- 接口前缀仍为 `/api/v1`，响应结构沿用主接口文档的统一响应结构。

## 1. 检查结论

| 类型 | 结论 |
| --- | --- |
| 前端 service 已存在但主接口文档漏写 | 用户修改、用户角色分配、用户删除 |
| 需求明确提到但主接口文档缺失 | 师生账号 Excel 批量导入预览与提交、教务基础数据导入预览与提交 |
| 需求提到但当前数据库或一期范围不支撑 | AI 引擎配置、课程年度质量报告固化管理 |
| 前端 service 与主接口文档冲突 | `/roles` 角色 CRUD service 存在，但主接口文档说明角色新增/修改/删除暂未开放 |

## 2. 用户管理接口补充

来源：

- 前端已有 `frontend/src/services/auth-service.ts`
- 主接口文档 `5.3 用户管理` 已定义查询、新增、状态、重置密码，但漏写修改、角色分配、删除
- 数据库支撑：`sys_user`、`sys_user_role`、`sys_role`、`student_info`

### 2.1 修改用户信息

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| `PUT` | `/api/v1/admin/users/{id}` | 修改用户基础信息 |

请求示例：

```json
{
  "realName": "张老师",
  "email": "teacher@example.com",
  "phone": "13800000000",
  "status": 1,
  "orgId": 10,
  "roleCodes": ["INSTRUCTOR"],
  "studentNo": null,
  "classId": null
}
```

实现说明：

- `roleCodes` 如传入，应同步维护 `sys_user_role`。
- 当 `roleCodes` 包含 `STUDENT` 时，`studentNo`、`classId` 必填，并同步维护 `student_info`。
- 当用户从学生角色变为非学生角色时，后端应按业务规则清理或忽略 `student_info`。

响应数据：同用户详情结构。

### 2.2 分配用户角色

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| `PUT` | `/api/v1/admin/users/{id}/roles` | 覆盖分配用户角色 |

请求示例：

```json
{
  "roleCodes": ["COORDINATOR", "INSTRUCTOR"]
}
```

字段说明：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `roleCodes` | string[] | 是 | 角色代码列表，仅允许系统预置角色 |

实现说明：

- 后端按 `roleCodes` 覆盖写入 `sys_user_role`。
- 若包含 `STUDENT`，应校验该用户存在 `student_info` 或在修改用户接口中补齐学生信息。

响应数据：

```json
true
```

### 2.3 删除用户

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| `DELETE` | `/api/v1/admin/users/{id}` | 删除用户 |

实现说明：

- 删除 `sys_user` 时，`sys_user_role` 按外键级联删除。
- `student_info` 按外键级联删除。
- `sys_audit_log` 不应物理删除，日志中冗余的 `username` 用于保留追溯信息。
- 若用户已被业务数据引用且后端不允许物理删除，可返回业务错误并建议前端改用禁用账号。

响应数据：

```json
true
```

## 3. 师生账号批量导入接口补充

来源：

- `系统角色功能表.md`：`1.1 师生账号批量导入`，要求 Excel 预览表字段包括 `工号/学号`、`姓名`、`角色`、`所属专业/班级`、`手机号`、`校验结果`
- `需求分析.md`：管理员负责学校教务基础数据同步，支持导入师生、行政班级及开课任务
- 主接口文档目前只写“用户由管理员开户或批量导入”，但没有定义用户导入接口
- 数据库支撑：`sys_user`、`sys_role`、`sys_user_role`、`sys_organization`、`class_info`、`student_info`

### 3.1 师生账号导入预览

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| `POST` | `/api/v1/admin/users/import/preview` | 上传 Excel 并返回账号导入预览和校验结果 |

请求格式：`multipart/form-data`

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `file` | File | 是 | `.xlsx` 或 `.xls` 文件 |

响应数据：

```json
{
  "batchId": "USER_IMPORT_20260720103000123",
  "totalRows": 120,
  "validRows": 116,
  "invalidRows": 4,
  "rows": [
    {
      "rowIndex": 2,
      "username": "T2024001",
      "realName": "张老师",
      "roleCodes": ["INSTRUCTOR"],
      "organizationName": "软件工程专业",
      "className": null,
      "phone": "13800000000",
      "validation": "PASS",
      "message": "校验通过"
    },
    {
      "rowIndex": 3,
      "username": "20240001",
      "realName": "李同学",
      "roleCodes": ["STUDENT"],
      "organizationName": "软件工程专业",
      "className": "软件工程 2401 班",
      "phone": "13900000000",
      "validation": "FAIL",
      "message": "行政班不存在"
    }
  ]
}
```

字段说明：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `batchId` | string | 预览批次 ID，用于提交导入 |
| `totalRows` | number | Excel 数据行数 |
| `validRows` | number | 校验通过行数 |
| `invalidRows` | number | 校验失败行数 |
| `rows` | object[] | 逐行预览结果 |
| `rows[].validation` | string | `PASS` 或 `FAIL` |
| `rows[].message` | string | 校验结果说明 |

校验规则：

- `username` 必填且不能与 `sys_user.username` 重复。
- `realName` 必填。
- `roleCodes` 必须属于系统预置角色。
- 教师类账号应能匹配所属组织。
- 学生类账号必须匹配行政班，并写入 `student_info`。
- 同一个 Excel 中不允许重复账号。

### 3.2 提交师生账号导入

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| `POST` | `/api/v1/admin/users/import` | 提交已通过预览校验的账号导入批次 |

请求示例：

```json
{
  "batchId": "USER_IMPORT_20260720103000123",
  "defaultPassword": "123456"
}
```

字段说明：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `batchId` | string | 是 | 预览接口返回的批次 ID |
| `defaultPassword` | string | 否 | 默认初始密码；不传时由后端使用系统默认值 |

响应数据沿用主接口文档 `14.1 Excel 导入响应`：

```json
{
  "totalRows": 120,
  "successRows": 116,
  "failedRows": 4,
  "errors": [
    {
      "rowIndex": 3,
      "field": "className",
      "message": "行政班不存在"
    }
  ]
}
```

实现说明：

- 成功行写入 `sys_user`。
- 按 `roleCodes` 写入 `sys_user_role`。
- 学生账号同步写入 `student_info`。
- 密码必须后端加密存储。
- 导入行为应写入 `sys_audit_log`。

## 4. 教务基础数据导入接口补充

来源：

- `需求分析.md`：管理员提供标准教务集成数据导入接口，一键导入和同步教师、学生、班级结构和排课计划
- `系统角色功能表.md`：`1.5 排课与选课关系导入`
- 主接口文档已有教学班 CRUD、教学班学生导入、成绩导入，但缺少面向教务排课/选课关系的一次性导入入口
- 数据库支撑：`sys_user`、`class_info`、`student_info`、`course`、`teaching_class`、`teaching_class_student`

一期建议先支持 Excel 导入；若后续接学校教务系统 API，可在该模块下扩展同步任务接口。

### 4.1 教务排课与选课导入预览

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| `POST` | `/api/v1/academic-imports/teaching-classes/preview` | 上传教务排课与选课 Excel，返回教学班和选课关系预览 |

请求格式：`multipart/form-data`

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `file` | File | 是 | 教务排课与选课 Excel |

响应数据：

```json
{
  "batchId": "TC_IMPORT_20260720104000123",
  "totalRows": 300,
  "validRows": 296,
  "invalidRows": 4,
  "teachingClassCount": 8,
  "studentRelationCount": 296,
  "rows": [
    {
      "rowIndex": 2,
      "semester": "2025-2026-1",
      "courseCode": "SE-302",
      "courseName": "软件工程",
      "teacherUsername": "T2024001",
      "teacherName": "张老师",
      "teachingClassName": "软件工程 01 班",
      "studentNo": "20240001",
      "studentName": "李同学",
      "studentClassName": "软件工程 2401 班",
      "validation": "PASS",
      "message": "校验通过"
    }
  ]
}
```

校验规则：

- `semester`、`courseCode`、`teacherUsername`、`teachingClassName`、`studentNo` 必填。
- `courseCode` 必须能匹配 `course`。
- `teacherUsername` 必须能匹配教师用户。
- `studentNo` 必须能匹配学生用户或根据导入策略提前通过账号导入创建。
- 同一教学班内同一学生不能重复。

### 4.2 提交教务排课与选课导入

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| `POST` | `/api/v1/academic-imports/teaching-classes` | 提交教务排课与选课导入批次 |

请求示例：

```json
{
  "batchId": "TC_IMPORT_20260720104000123"
}
```

响应数据沿用主接口文档 `14.1 Excel 导入响应`。

实现说明：

- 按 `semester`、`courseId`、`teacherId`、`className` 创建或更新 `teaching_class`。
- 写入 `teaching_class_student` 建立选课关系。
- 不在该接口中创建课程；课程应先由课程接口或课程导入接口维护。
- 不在该接口中创建学生账号；学生账号应先由账号导入或用户管理接口维护。
- 导入行为应写入 `sys_audit_log`。

## 5. 暂不追加项

以下内容在检查中发现，但不作为本次正式追加接口。

### 5.1 角色 CRUD 接口

现状：

- `frontend/src/services/auth-service.ts` 中存在 `/roles` 的查询、新增、修改、删除 service。
- 主接口文档 `5.2 角色基础数据` 明确说明：当前系统角色为预置基础数据，角色新增、修改、删除接口暂未开放。

结论：

- 不追加 `/api/v1/roles` CRUD 为正式接口。
- 前端后续接真实服务时，应使用 `/api/v1/dicts/roles` 获取角色枚举，管理员只做“分配角色”，不做“管理角色”。
- 建议后续清理或停用前端 `/roles` CRUD service，避免误接。

### 5.2 AI 引擎配置接口

现状：

- `需求分析.md` 中系统管理员定位包含 AI 引擎配置。
- `数据库.md` 曾包含 `ai_prompt_template`。
- `数据库变更记录.md` 明确说明 AI 提示词配置表已移除，版本一期暂不实现 AI Agent 集成。

结论：

- 当前数据库变更记录已移除支撑表，不追加 AI 配置接口。

### 5.3 课程年度质量报告接口

现状：

- `需求分析.md` 和 `系统角色功能表.md` 提到课程级年度质量报告。
- 当前数据库没有独立课程年度报告表。
- 现有 `report`、`report_section`、`report_data_source` 面向专业认证自评报告，不等价于课程年度质量报告。

结论：

- 不追加课程年度质量报告 CRUD。
- 若要实现，需要先确认是复用自评报告模块，还是新增课程报告表。

### 5.4 审计日志导出接口

现状：

- 主接口文档只定义审计日志分页查询和详情。
- `系统角色功能表.md` 对审计日志描述为分页查询、只读不可篡改，没有明确导出。

结论：

- 不追加审计日志导出接口。
